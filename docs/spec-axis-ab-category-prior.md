# Spec:平台分区作轴 A/B 分类先验

> 状态:**已实现并合入**(commit `46a61dc`,2026-07-14)。本文件是追溯式设计记录(design record / PRD),捕获 grill 定案的决策与依据;规范正文(what)见 `cuepoint-spec-v0.5.md` §5.1/§5.2,本文件补的是「为什么这么定」(why)。
> 性质:**纯文档改动**——落在 `SKILL.md` 步骤 3 与 spec §5.1/§5.2,`fetch.py`/schema/`scripts/*`/测试一行未动,`video_index.json` 契约不变。

## Problem Statement

宿主 Claude 在步骤 3 判定视频的轴 A(`visual_form` 画面形态)与轴 B(`genre` 内容体裁)时,轴 B 只吃 transcript + 标题 + 简介 + chapters,**把 yt-dlp 已经抓到、就躺在 `.work/ytdlp_info.json` 里的平台分区信号白白丢弃**(`fetch.py:parse_metadata` 只保留 title/duration/language/uploader)。用户希望借 B 站分区体系与 YouTube category 体系,把 A/B 分类判得更准。

同时存在一处历史 drift:体裁集合在 spec §5.2 列 8 项(无「直播实录/秀场」),而 `SKILL.md` 步骤 3/8 列 9 项——两处不一致,是刚清完 stale 状态后又冒出的一处新分叉。

## Solution

把平台分区(B 站分区 / YouTube `categories`+`tags`)接进步骤 3,当作**软提示**参与轴 A/B 判定,而**不是**新增分类桶、也**不是**建硬映射表:

- **轴 B**:分区与 transcript/标题同权,作为体裁判定的一个额外输入。明确「分区=主题域(讲什么)、体裁=生产形态(哪种片)」二者正交——一个分区可对多种形态(美食可以是教程/vlog/评测),故不建「分区→体裁」对照表。
- **轴 A**:分区作**不对称**先验——**只去锚、不定性**。演出/虚构/二创类分区(番剧/电影/音乐/舞蹈/游戏等)是强「非 slide-driven」信号,用来把判断推离 slide-driven;但永不得单凭分区把视频定成 slide-driven,定性仍由探针 sheet 目验独占。
- **非目标降级**:判为演出/虚构非目标时如实告知用户会降级,但遵守零打断契约,提醒并入渲染前唯一交互点。
- 顺手把体裁集合从 8 对齐到 9,消除 SKILL⇄spec drift。

用户由此:更准的体裁→风格/骨架路由,更少的假 slide-driven 误判,以及对「本片不是本工具甜区」的诚实预期。

## User Stories

1. 作为跑本 skill 的宿主 Claude,我想在判定轴 B 体裁时读到平台分区,以便在 transcript 语义模糊时多一个佐证输入,把体裁判得更准。
2. 作为宿主 Claude,我想知道分区是「软提示」而非「硬映射」,以便当分区与 transcript 语义冲突时,放心地以语义为准而不被分区带偏。
3. 作为宿主 Claude,我想在判定轴 A 时用分区把判断**推离** slide-driven(当分区是音乐/番剧/游戏等演出/虚构类),以便少犯「把 cinematic/录屏误判成 slide-driven」的错。
4. 作为宿主 Claude,我想被明确禁止「单凭分区把视频定成 slide-driven」,以便不重蹈 2026-07-11 批量试产里「数值先验锚定假 slide」的覆辙。
5. 作为宿主 Claude,我想在分区+目验判为演出/虚构非目标时提醒用户工具会降级,以便用户对帧退化为氛围/装饰有预期。
6. 作为宿主 Claude,我想把上述降级提醒并入渲染前唯一交互点、而非分析中途弹出,以便守住「分析全程零打断」契约。
7. 作为宿主 Claude,我想在读分区时对缺失 fail-open,以便本地文件、以及 B 站(`categories` 恒空)这些无分区场景下主流程不中断。
8. 作为终端用户,我想让工具在字幕驱动内容上判得更准,以便产出的大纲骨架与 slide 风格更贴合视频真实体裁。
9. 作为终端用户,我想在上传一个本工具不擅长的视频(演出/虚构类)时被诚实告知会降级,以便决定是否继续渲染或换片。
10. 作为终端用户,我不想在分析过程中被任何弹窗打断,以便一次交互点里同时决定形态/粒度/语言与是否继续。
11. 作为 video_index 文档的第三方消费者,我想让分区只影响分析期判断、不进导出契约,以便 `video_index.json` 的 schema 稳定不变。
12. 作为未来的维护者,我想让分区信号以散文契约(读 `ytdlp_info.json`)而非新代码路径落地,以便不为一个软提示新增维护面。
13. 作为未来的维护者,我想让 spec 与 SKILL 的体裁集合对齐到 9 项,以便不再有 SKILL⇄spec drift 需要复核。
14. 作为未来的维护者,我想在 spec 里看到「为什么不建硬表」的实测反例,以便有人再提「加张分区对照表」时能被历史挡回去。
15. 作为设计评审者,我想在文档里看到「分区(主题域)与体裁(生产形态)正交」的明确表述,以便不会有人把两个轴压成一个、重引组合矩阵。
16. 作为跨平台用户,我想让方案在 YouTube(`categories` 稳定)与 B 站(`categories` 恒空、只有噪声 `tags`)两侧都成立,以便同一套流程覆盖两平台。

## Implementation Decisions

- **信号定位(b-软)**:平台分区作**软提示**,不建「分区→genre」硬映射表。理由:平台分类法又大又漂(B 站主+二级约 40 项、YouTube 约 15 项、`tags` 开放集),硬表是长期 stale 源;轴 B 本就是 Claude 语义判断步骤,多一个同质输入即可;分区与体裁非 1:1。
- **落地形态(b-文字)**:纯改 `SKILL.md` 步骤 3 散文,让宿主连带读 `ytdlp_info.json` 的 `categories`/`tags`(该文件步骤 3 已为读简介而打开)。**不改 `fetch.py`、不进 `meta.json`、不进 schema**。理由:与 b-软 极简一致;fail-open 更干净(本地文件与 B 站天然缺该字段);不污染对外契约。
- **轴 A 用法(A-非对称)**:分区对轴 A **只能去锚、不能定性**——可把判断推离 slide-driven,但永不得单凭分区定为 slide-driven,目验是唯一定性者。理由:对称弱先验会把「分区=知识/教程」锚向 slide-driven 触发确认偏误(即 2026-07-11 那道疤);不对称只保留纠错方向(批量试产的假 slide-driven #3/#6/#18 全落在「误判为 slide」一侧),不开反向锚定口子。
- **桶集合(a-0)**:**不新增任何体裁桶**。理由:轴 B 每加一桶要连带定一套大纲骨架 + 选一个风格预设;平台那些「多出来的」分区压倒性地要么折进现有形态(科技数码/知识/美食→课程/评测/vlog…),要么是演出/虚构非目标(番剧/电影/音乐/游戏)。力气花在磨锋利(非目标信号 + 定义澄清)ROI 更高。
- **修 drift**:体裁集合 spec §5.2 由 8 → 9,补「直播实录/秀场」,对齐 SKILL 步骤 3/8。
- **改动范围(改-双)**:`SKILL.md` 作操作契约(主改)+ spec §5.1/§5.2 同步,避免只改 SKILL 造成 source-of-truth 立即 stale。
- **正交契约**:文档明确记「分区=主题域、体裁=生产形态,二者正交,不枚举组合矩阵」,与既有两轴正交消费原则一致。

## Testing Decisions

- **本改动无代码 seam**——它是对 LLM 编排契约(SKILL.md prose)+ 其 spec 的改动,不存在可单元测试的函数边界。好的验证在此处是「外部行为」层面的:**用真实视频跑 `yt-dlp` 元数据 dump,核对分区信号实际落在哪个字段、是否可靠**,而非 mock。
- **已执行的实测 seam**(yt-dlp 2026.07,样本 3 YouTube + 4 B 站):
  - YouTube `categories` 稳定给**单个干净值**(`Education` / `Science & Technology` / `News & Politics`,3/3)→ 可直接当体裁软提示。
  - B 站 `categories` **恒空**(4/4),官方分区(tname)yt-dlp 不透出;唯一沾边的 `tags` 有噪声、≠ 分区,实例:①汽车区 UP 的酒店 vlog `tags` 仍是 `['阿B车友俱乐部']`;②足球分析视频 `tags` 首位被 `淘宝闪购…` 广告标签占满。这两例同时**反证了不建硬表**:软提示可被 transcript 语义覆盖,硬表会被误路由。
- **验证覆盖的模块**:步骤 3 判定行为(宿主读取分区并正确降权/去锚)、`fetch.py` 落盘的 `ytdlp_info.json` 确含该字段。
- **回归判据**:轴 A 判定继续经探针 sheet 目验定性(现行 SKILL 机制不变),分区不得使其绕过目验——即本改动**不放松**已有的「数值判据不可独立成立」硬规则。
- **Prior art**:测试源与实测结论登记在 `docs/test-videos.md`(27 个已验证测试源 + 批量试产发现),本轮 7 条 dump 结论同源同格式沉淀。

## Out of Scope

- 把平台分区升格为 `meta.json` 结构化字段或写进 `video_index.json` 契约(YAGNI,目前只有步骤 3 一个消费者)。
- 建任何「分区→体裁」或「分区→visual_form」对照表(已明确否决)。
- 新增体裁桶(娱乐/综艺、游戏实况、影视解说等)——本轮 a-0 决定不加。
- 为拿到 B 站真实分区(tname)而绕开 yt-dlp 直连 B 站 web API(当前架构 fetch 层只 shell out `yt-dlp` CLI,改这条是架构变更,非本 spec)。
- `import yt_dlp`(违反 fetch 层只走 CLI subprocess 的不变量)。
- YouTube 侧 `tags` 的深加工(实测常为空或话题词,当前不依赖)。
